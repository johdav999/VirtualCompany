# Goal
Implement backlog task **TASK-11.5.5** for **ST-505 Daily briefings and executive summaries** by delivering the first usable slice of proactive summaries: **scheduled in-app generation only**. Do **not** implement email delivery in this task.

The implementation should enable the system to:
- generate scheduled **daily briefings** and **weekly summaries** per company,
- persist them as in-app communication artifacts,
- make them visible to the dashboard/inbox surfaces through existing or minimally extended backend contracts,
- support tenant-safe background execution,
- leave clean extension points for future mobile notification and email delivery.

This task should align with the architecture:
- modular monolith,
- ASP.NET Core backend,
- PostgreSQL persistence,
- background workers for scheduled jobs,
- outbox/reliable side effects where appropriate,
- tenant-scoped execution and storage.

Because no explicit acceptance criteria were provided for the task itself, use the parent story ST-505 and the note **“Start with scheduled in-app generation; email delivery can follow.”** as the implementation boundary.

# Scope
In scope:
- Add backend support for scheduled generation of:
  - daily briefings,
  - weekly executive summaries.
- Store generated summaries as **messages and/or notifications** suitable for in-app display.
- Ensure summaries are **company-scoped** and generated by a **background worker/scheduler path**, not on-demand only.
- Aggregate summary inputs from existing domain data where available, prioritizing:
  - alerts/exceptions,
  - pending approvals,
  - KPI highlights if accessible,
  - notable agent/task/workflow updates.
- Add minimal configuration support needed to determine whether in-app delivery is enabled and when generation should occur.
- Add application/query support so web dashboard surfaces can retrieve the latest generated briefing/summary.
- Add tests for scheduling/generation/persistence behavior.

Out of scope:
- Email delivery.
- Push notification delivery.
- Full user-facing preferences center if not already present.
- Rich LLM prompt optimization beyond a pragmatic first version.
- New mobile-specific implementation.
- Large dashboard redesign.
- Complex per-user personalization if company-level generation is the current simplest fit.

If the codebase lacks some supporting primitives, implement the smallest coherent vertical slice that preserves future extensibility.

# Implementation plan
1. **Inspect current architecture and existing primitives**
   - Find existing implementations for:
     - background jobs / hosted services / schedulers,
     - `messages`, `conversations`, and `notifications`,
     - task/workflow/approval query models,
     - tenant resolution and company-scoped repositories,
     - audit/outbox patterns.
   - Reuse existing patterns rather than inventing a parallel mechanism.

2. **Define the minimal domain model for scheduled in-app summaries**
   - Prefer extending existing communication models over creating a brand-new subsystem.
   - If `messages` already supports `message_type`, add or use values such as:
     - `daily_briefing`
     - `weekly_summary`
     - or reuse `summary` with structured metadata indicating cadence/type.
   - If notifications exist separately, decide whether:
     - the generated summary is stored as a message and optionally mirrored as a notification, or
     - the summary is stored in a dedicated notification payload.
   - Favor a design where the full summary content lives in a durable message-like record, with notifications as lightweight pointers.

3. **Add scheduling/configuration support**
   - Implement the smallest practical configuration model for ST-505:
     - company-level enablement for in-app daily briefings,
     - company-level enablement for in-app weekly summaries,
     - schedule timing based on company timezone.
   - If there is already a company settings JSON structure, extend it there rather than introducing a new table unless a dedicated table is clearly already the project convention.
   - Suggested shape:
     - `briefings.daily.enabled`
     - `briefings.daily.localTime`
     - `briefings.weekly.enabled`
     - `briefings.weekly.dayOfWeek`
     - `briefings.weekly.localTime`
     - `briefings.delivery.inApp`
   - Default to conservative, explicit behavior.

4. **Implement summary aggregation service**
   - Create an application service that gathers company-scoped inputs for a time window.
   - For daily briefings, aggregate the last 24 hours or previous local-day window.
   - For weekly summaries, aggregate the previous local-week window.
   - Include, where available:
     - pending approvals count and top items,
     - failed/blocked workflows or tasks,
     - notable completed tasks,
     - agent activity highlights,
     - KPI highlights from existing analytics aggregates if accessible,
     - anomalies/alerts if such records already exist.
   - Keep this deterministic and testable.
   - Return a structured model with:
     - title,
     - summary text,
     - sections,
     - linked entity references.

5. **Implement generation path**
   - If the system already has an AI orchestration/summarization service, use it only as a formatting/summarization layer on top of structured aggregates.
   - If not, generate a deterministic first-pass summary in code from the aggregate model.
   - Do not block delivery of this task on sophisticated LLM generation.
   - Persist:
     - summary body,
     - structured payload with sections and references,
     - sender type `system`,
     - company scope,
     - timestamps,
     - cadence metadata (`daily` / `weekly`),
     - covered time window.
   - Ensure idempotency so the same company/cadence/window is not generated multiple times.

6. **Implement scheduler/background worker**
   - Add or extend a background worker that periodically scans companies and determines whether a daily or weekly summary is due based on:
     - company timezone,
     - configured local schedule,
     - last generated window.
   - Use distributed locking / safe coordination if the project already uses Redis or another lock mechanism.
   - Ensure tenant-scoped execution and correlation IDs in logs.
   - If the codebase uses outbox for side effects, follow that pattern for any notification fan-out.

7. **Persist and expose in-app artifacts**
   - Store the generated summary in the communication layer so it is visible in-app.
   - If there is a dashboard “latest briefing” concept, add a query handler to fetch:
     - latest daily briefing,
     - latest weekly summary,
     - recent briefing history.
   - If notifications exist, create a lightweight notification indicating a new briefing is available, but only if this is low-friction and consistent with current architecture.
   - Do not add email dispatch.

8. **Add API/query surface**
   - Add or extend endpoints/query handlers for dashboard consumption.
   - Keep them tenant-scoped and authorization-safe.
   - Suggested capabilities:
     - get latest daily briefing for current company,
     - get latest weekly summary for current company,
     - list recent summaries.
   - Reuse existing CQRS-lite patterns.

9. **Add minimal web visibility**
   - If the web dashboard already has a daily briefing area, wire it to the new query.
   - Otherwise add a minimal, non-invasive dashboard card or section showing the latest generated briefing.
   - Keep UI changes small and aligned with existing patterns.
   - Do not overbuild preferences UX unless already scaffolded.

10. **Add tests**
   - Unit tests:
     - aggregation logic for daily and weekly windows,
     - schedule due-calculation with timezone handling,
     - idempotency behavior.
   - Integration tests:
     - generated summary persists to the correct company only,
     - latest briefing query returns expected records,
     - duplicate scheduler runs do not create duplicate summaries for the same window.
   - If feasible, test that disabled schedules do not generate summaries.

11. **Migration and documentation**
   - Add any required database migration following repository conventions.
   - Update relevant README or internal docs if there is a section for background jobs or briefing behavior.

Implementation guidance:
- Prefer **company-level scheduled generation** first, even if delivery preferences in the story mention users. If user-level preferences already exist, integrate with them; otherwise keep this slice company-scoped and in-app.
- Prefer **existing message/notification infrastructure** over new bespoke tables.
- Prefer **deterministic summaries** over delaying for full LLM orchestration.
- Ensure **tenant isolation**, **idempotency**, and **timezone correctness**.

   - confirm failures are logged safely and are retryable if applicable.

# Risks and follow-ups
Risks:
- The codebase may not yet have a mature scheduler abstraction; avoid overengineering and fit into existing hosted service patterns.
- Timezone and “once per window” logic can easily create duplicate or missed summaries.
- If analytics/alerts data is not yet implemented, summary content may need to rely mainly on tasks, approvals, and workflow state.
- If notifications and messages are both partially implemented, choosing the wrong persistence anchor could create rework; prefer durable message storage with optional notification pointer behavior.
- If user-level delivery preferences are not yet modeled, forcing them now may expand scope too much.

Follow-ups after this task:
- Add email delivery via outbox-backed dispatcher.
- Add mobile notification fan-out.
- Add user-level delivery preferences and digest subscriptions.
- Improve summary quality with LLM-assisted narrative generation over structured aggregates.
- Add richer dashboard history and filtering for briefings/summaries.
- Add explicit audit events for briefing generation and source references.
- Add caching of expensive aggregate inputs to reduce generation cost.
- Add admin UI for configuring briefing schedules and delivery preferences if not included here.
