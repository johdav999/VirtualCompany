# Goal
Implement backlog task **TASK-11.5.3** for **ST-505 Daily briefings and executive summaries** so that **generated summaries are persisted as messages and/or notifications and are visible in the dashboard**.

This task should extend the existing daily/weekly summary flow by ensuring that when a briefing/executive summary is generated for a company:

- it is stored durably in the communication/notification model,
- it can be queried for dashboard display,
- it remains tenant-scoped,
- it is suitable for later mobile/inbox reuse.

Prefer a minimal, production-aligned implementation that fits the current modular monolith and CQRS-lite architecture.

# Scope
In scope:

- Persist generated briefing/summary output into the existing domain model using:
  - `messages` for dashboard-visible briefing content, and
  - `notifications` only if a notification model already exists in the codebase or can be added with low friction and clear value.
- Add or extend application-layer command/query handlers to:
  - create stored summary records,
  - retrieve the latest summary for dashboard use.
- Expose API endpoints or internal application services needed by the web dashboard.
- Update the dashboard UI to surface the latest stored summary.
- Add tests covering persistence, tenant scoping, and dashboard retrieval.

Out of scope unless already partially implemented:

- Full scheduling engine for daily/weekly generation.
- Email delivery.
- Mobile UI changes.
- Rich notification preferences UX.
- Large refactors across unrelated modules.

Implementation guidance:

- If the codebase already has a `Conversation`/`Message` model, use a dedicated conversation/channel pattern for company briefings.
- If notifications already exist, create a lightweight notification entry linked to the stored summary; otherwise do not invent a large notification subsystem just for this task.
- Keep the summary body concise and user-facing. Do not store chain-of-thought.

# Implementation plan
1. **Inspect current communication and dashboard implementation**
   - Find existing entities, handlers, and UI for:
     - conversations
     - messages
     - notifications/inbox
     - dashboard widgets/cards
     - scheduled or generated summaries
   - Determine whether ST-505 already has partial generation logic that currently returns transient output without persistence.

2. **Choose the persistence shape**
   - Prefer this model if compatible with the codebase:
     - create or reuse a company-scoped conversation for briefings, e.g. channel type like `briefing` or `system_summary`
     - store each generated daily/weekly summary as a `Message`
       - `sender_type = system` or `agent`
       - `message_type = summary`
       - `body = rendered summary text`
       - `structured_payload = JSON` containing metadata such as:
         - summary kind (`daily` / `weekly`)
         - generated timestamp
         - source references (task IDs, approval IDs, KPI refs) if available
   - If a notification entity already exists:
     - create a notification record pointing to the stored message/summary for dashboard/inbox visibility.
   - Do not create duplicate business concepts unless needed.

3. **Add domain/application support for storing summaries**
   - Implement a command or application service such as:
     - `StoreGeneratedSummaryCommand`
     - or `CreateCompanyBriefingMessageCommand`
   - Inputs should include:
     - `CompanyId`
     - summary type
     - summary body
     - optional structured payload/source references
     - generated by actor/system metadata
   - Ensure tenant ownership is enforced.

4. **Handle conversation creation/reuse**
   - If summaries are stored as messages, ensure there is a deterministic way to find or create the company’s briefing conversation.
   - Recommended behavior:
     - one company-level conversation per summary channel/type, reused over time
     - latest summary is the most recent `summary` message in that conversation
   - Avoid creating a new conversation for every summary unless existing patterns require it.

5. **Add dashboard retrieval query**
   - Implement a query such as:
     - `GetLatestDashboardBriefingQuery`
     - or extend an existing dashboard query response
   - Return:
     - summary id/message id
     - summary type
     - body
     - created/generated timestamp
     - optional source links/counts
   - Query must be tenant-scoped and efficient.

6. **Expose API surface**
   - If dashboard data is served through API endpoints, add/extend the endpoint to include the latest stored summary.
   - Follow existing auth and company-context resolution patterns.
   - Return empty/null state cleanly when no summary exists.

7. **Update dashboard UI**
   - Add or update the dashboard card/widget to display the latest stored briefing.
   - Show:
     - title
     - generated time
     - summary body
     - optional “view details” or related links if already supported
   - Include an empty state when no summary has been generated yet.

8. **Schema changes if required**
   - Only add schema changes if the current model cannot support this cleanly.
   - Possible minimal additions:
     - new conversation channel type enum/string usage
     - indexes for latest-summary lookup
     - notification table only if the product already expects one and it is missing
   - Use the repo’s established migration approach.

9. **Testing**
   - Add tests for:
     - storing a generated summary creates a message under the correct company
     - repeated summaries reuse the expected conversation if applicable
     - dashboard query returns the latest summary only for the active tenant
     - unauthorized cross-tenant access is blocked or returns not found/forbidden per existing conventions
     - dashboard UI/API handles no-summary state

10. **Keep implementation aligned with architecture**
   - Respect modular boundaries:
     - no UI-layer persistence logic
     - no direct DB access from web components
   - Keep this as CQRS-lite:
     - command for persistence
     - query for dashboard projection
   - Preserve auditability through structured payload/source references where feasible.

# Risks and follow-ups
- **Notification model ambiguity:** The backlog says “messages/notifications.” If notifications are not yet implemented, forcing a new subsystem here may create unnecessary scope. Prefer messages-first unless notification infrastructure already exists.
- **Conversation model mismatch:** The repo may not yet model company-level system conversations for briefings. Reuse existing patterns rather than inventing a parallel structure.
- **Dashboard coupling:** Avoid embedding summary generation logic into the dashboard query. This task is about persisted visibility, not on-demand generation.
- **Schema drift:** If enums/constants for channel/message types are hardcoded in multiple layers, update them carefully to avoid serialization/query bugs.
- **Future follow-up:** A later task should connect scheduled generation, delivery preferences, inbox/mobile notification fan-out, and possibly unread/read state for briefing availability.
